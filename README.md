# Healthcare Data Warehouse using Python, Airflow, dbt & PostgreSQL

An end-to-end modern data engineering project that builds a production-style Healthcare Data Warehouse using **Python**, **PostgreSQL**, **Apache Airflow**, **dbt**, and **Docker**.

The project ingests raw healthcare datasets, loads them into a PostgreSQL staging layer, transforms them into an analytics-ready dimensional warehouse using dbt, performs automated data quality testing, captures operational metadata, and orchestrates the entire workflow through Apache Airflow.

# Project Overview

Healthcare organizations generate massive volumes of operational data every day, including patient registrations, appointments, treatments, billing records, doctor information, and hospital data. These datasets often originate from multiple operational systems and require consolidation before they can support reporting and analytics.

This project demonstrates how modern data engineering tools can be combined to build a production-inspired healthcare analytics platform.

The pipeline performs the following tasks:

- Reads multiple healthcare CSV datasets
- Loads raw data into PostgreSQL staging tables
- Applies dimensional modeling using dbt
- Creates reusable fact and dimension tables
- Executes automated data quality tests
- Records pipeline execution metadata
- Captures table-level operational metrics
- Orchestrates the complete workflow using Apache Airflow
- Runs inside a fully containerized Docker environment

# Pipeline Workflow

The Healthcare Data Warehouse pipeline follows a layered architecture commonly used in modern analytics platforms.

### Step 1 — Raw Data Ingestion

The pipeline begins by reading six healthcare datasets from CSV files.

- Patients
- Doctors
- Hospitals
- Appointments
- Treatments
- Billing

Each dataset is loaded using Pandas before being validated and inserted into PostgreSQL staging tables.

---

### Step 2 — Staging Layer

The staging schema acts as the landing zone for raw operational data.

During every pipeline execution:

- Existing staging tables are truncated
- Latest CSV data is loaded
- Table-level metrics are collected
- Execution time is recorded
- Row counts are captured

This layer preserves the raw structure of incoming data before any business transformations are applied.

---

### Step 3 — Data Transformation using dbt

After loading completes successfully, Apache Airflow triggers dbt.

dbt performs:

- Data cleaning
- Standardization
- Business rule implementation
- Surrogate key generation
- Derived column creation
- Fact table creation
- Dimension table creation

The final analytics layer follows a Star Schema design.

---

### Step 4 — Data Quality Validation

Once transformations finish, dbt executes automated data quality tests.

The pipeline validates:

- Primary Keys
- Unique Constraints
- Null Checks
- Accepted Values
- Foreign Key Relationships

If any validation fails, the DAG fails immediately, preventing invalid data from reaching downstream consumers.

---

### Step 5 — Pipeline Monitoring

Every execution generates operational metadata including:

- Pipeline status
- Start time
- End time
- Duration
- Rows processed
- Airflow Run ID
- Error messages (if any)

Additional table-level metrics are also recorded.

# Warehouse Design

The warehouse follows a classic dimensional modeling approach.

```
                Fact Appointments
                       │
      ┌────────────────┼────────────────┐
      │                │                │
      ▼                ▼                ▼
 Dim Patient     Dim Doctor      Dim Hospital
      │
      ▼
   Dim Date
```

The dimensional model enables efficient analytical queries while minimizing redundancy.

# Apache Airflow Workflow

Apache Airflow orchestrates the complete pipeline.

The DAG consists of three primary tasks.

```
Extract & Load Raw Data
          │
          ▼
Transform Models (dbt Run)
          │
          ▼
Run Data Quality Tests
```

### Task 1 — Extract and Load

Python ETL performs:

- CSV ingestion
- PostgreSQL loading
- Pipeline auditing
- Metrics collection

---

### Task 2 — dbt Run

Executes:

```bash
dbt deps
dbt run
```

This builds the analytics warehouse.

---

### Task 3 — dbt Test

Executes:

```bash
dbt test
```

All data quality rules are validated before the DAG completes successfully.

# Data Quality Framework

Data quality is enforced automatically through dbt tests.

The pipeline currently validates:

## Uniqueness Tests

Ensures:

- Patient IDs are unique
- Doctor IDs are unique
- Hospital IDs are unique
- Appointment IDs are unique
- Billing IDs are unique
- Treatment IDs are unique

---

## Not Null Tests

Critical business fields cannot contain null values.

Examples include:

- Patient ID
- Doctor ID
- Appointment ID
- Hospital ID
- Billing Amount
- Treatment Cost

---

## Accepted Values Tests

Ensures only valid business values exist.

Examples:

Gender

```
Male
Female
```

Appointment Status

```
Scheduled
Completed
Cancelled
```

Hospital Type

```
Government
Private
Teaching
Specialty
```

Doctor Experience

```
Junior
Mid
Senior
```

---

## Relationship Tests

Foreign key integrity is verified between tables.

Examples:

- Appointment → Patient
- Appointment → Doctor
- Appointment → Hospital
- Billing → Appointment
- Treatment → Appointment

These tests ensure no orphan records exist within the warehouse.

# Pipeline Audit

Every Airflow execution records metadata in the `metadata.pipeline_audit` table.

The audit framework captures:

- Pipeline name
- Airflow Run ID
- Pipeline status
- Start time
- End time
- Execution duration
- Total rows processed
- Error message (if any)

Example:

| Audit ID | Status | Rows Loaded | Duration (sec) |
|-----------|---------|-------------|---------------:|
| 40 | SUCCESS | 310,550 | 4.12 |
| 39 | SUCCESS | 310,550 | 4.05 |

This provides complete traceability for every pipeline execution and simplifies operational monitoring.

---

# Table Load Metrics

In addition to pipeline-level auditing, the project records metrics for every table loaded into the staging layer.

Each table load captures:

- Table name
- Number of rows loaded
- Load duration
- Pipeline name
- Timestamp

Example:

| Table | Rows | Load Time (sec) |
|--------|-----:|----------------:|
| patients | 10,000 | 0.27 |
| appointments | 100,000 | 2.71 |
| billing | 100,000 | 2.38 |
| doctors | 500 | 0.17 |
| hospitals | 50 | 0.08 |
| treatments | 100,000 | 1.46 |

These metrics help identify performance bottlenecks and monitor ingestion throughput over time.
